# 혜화 데이터랩 프로젝트용 CLAUDE.md

이 프로젝트는 **혜화 데이터랩 스타일의 데이터 리터러시 수업자료**를 만드는 저장소다.

Claude Code는 이 저장소에서 단순한 코드 생성기가 아니라, **기존 lesson 패턴을 읽고 같은 흐름과 톤으로 새 수업자료를 제작하는 교재 설계 도우미**처럼 행동해야 한다.

## 이 저장소의 실제 구현 기준

- 이 프로젝트는 **React + Vite + Tailwind CSS** 기반으로 작업한다.
- 패키지 매니저는 **npm** 기준으로 생각한다.
- 기본 실행 명령은 다음을 기준으로 삼는다.
  - 개발 서버: `npm run dev`
  - 빌드 확인: `npm run build`
  - 미리보기: `npm run preview`
- 현재 별도의 lint/test 스크립트가 없더라도, 코드 품질과 구조 일관성은 스스로 점검한다.

## 이 프로젝트의 구조 규칙

- 모든 lesson 메타데이터는 `src/data/lessonRegistry.js`를 기준으로 관리한다.
- 새 lesson을 추가할 때는 **lesson 파일만 만들지 말고 반드시 registry도 함께 갱신**한다.
- lesson 컴포넌트는 아래 규칙을 따른다.
  - 위치: `src/lessons/moduleX/`
  - 파일명: `LessonX_Y.jsx`
  - 예시: `src/lessons/module2/Lesson2_4.jsx`
- lesson id `X-Y`는 파일 경로 `moduleX/LessonX_Y.jsx`와 일치해야 한다.
- 홈 화면, 사이드바, 이전/다음 lesson 흐름은 `lessonRegistry`를 기준으로 반영된다고 생각하고 작업한다.
- 새 lesson을 넣을 때는 가능하면 **홈, sidebar, navigation에서 자연스럽게 이어지는지 함께 확인**한다.

## 전용 서브에이전트 우선 사용

다음 작업에서는 **반드시 먼저** `.claude/agents/hyehwa-course-ui-builder.md` 서브에이전트를 우선 사용한다.

- 홈 화면 제작 또는 개편
- 새 module / 새 lesson 추가
- 기존 lesson 리팩터링
- 실습 블록 강화
- sidebar / lessonRegistry / lesson navigation 수정
- 공통 lesson UI 컴포넌트 설계

이 프로젝트의 공통 규칙은 이 `CLAUDE.md`를 따르고, 실제 화면/lesson 제작은 위 서브에이전트를 우선 활용한다.

## 작업 시작 전에 먼저 할 일

- 현재 저장소의 라우팅, lessonRegistry, Sidebar, LessonNav, Home 구조를 먼저 파악한다.
- 새 lesson을 만들기 전에는 **기존 lesson들의 실습 방식과 상호작용 패턴**을 먼저 조사한다.
- 특히 사이드바에 있는 여러 lesson의 실습 블록을 참고해, 이 프로젝트에 이미 어울리는 활동 문법을 파악한다.
- 새 콘텐츠를 만들 때는 기존 문장을 복붙하지 말고, **패턴만 재사용하고 내용은 새로 쓴다.**

## 수업 설계 원칙

- 이 프로젝트에는 고정된 섹션 템플릿이 없다.
- `개념`, `실습`, `생각해보기`, `정리`, `퀴즈`, `미니 활동`, `되짚기` 같은 섹션은 **lesson 주제에 맞게 유동적으로 구성**한다.
- 섹션 순서도 고정하지 않는다.
- 다만 전체 흐름에서는 **개념과 실습이 섞여 있는 리듬**을 유지한다.
- 같은 프로젝트 안에서 모든 lesson이 똑같은 틀을 반복하지 않도록 한다.
- 설명만 길게 이어지는 정적 페이지보다, 학생이 중간중간 반응하고 생각할 수 있는 lesson을 우선한다.
- 한 lesson 안에서 다음과 같은 흐름은 자연스럽게 허용된다.
  - 도입 → 개념 → 짧은 실습 → 개념 심화 → 생각해보기 → 정리
  - 도입 → 실습 → 개념 설명 → 실습 → 되짚기
  - 질문 던지기 → 선택하기 → 이유 보기 → 개념 연결 → 확장 활동

## 실습 설계 원칙

- 실습 유형은 고정하지 않는다. **주제에 가장 잘 맞는 방식**을 직접 선택한다.
- 실습은 장식이 아니라 학습의 중심이어야 한다.
- 학생이 클릭, 선택, 비교, 분류, 예측, 작성, 답 확인 등의 행동을 하게 만드는 구성을 선호한다.
- 답안 공개 방식도 각 실습에 맞게 정한다.
  - 바로 피드백
  - 생각한 뒤 답 보기
  - 힌트 → 답 → 해설
  - 카드 뒤집기
  - 선택 후 이유 확인
  - 직접 써본 뒤 예시 답 보기
- 실습은 가능하면 학생 주변 맥락을 활용한다.
  - 학교생활
  - 급식
  - 통학
  - 교실 환경
  - 스마트폰/SNS
  - 수면, 운동, 공부 습관
  - 편의점, 날씨, 동아리, 용돈 등
- `우리 주변의 데이터` 같은 lesson의 감각을 참고해, 학생이 **"아 이건 내 이야기다"**라고 느낄 수 있는 예시를 우선한다.
- 막연한 `생각해보세요`보다, 판단 기준이나 선택지나 힌트를 주는 활동을 선호한다.
- 반응형 레이아웃뿐 아니라, **학생 행동에 반응하는 인터랙션**도 적극적으로 활용한다.

## 콘텐츠 톤과 대상

- 대상은 **고등학교 2~3학년 여학생**을 기본으로 생각한다.
- 한국어로 작성한다.
- 말투는 **친근하고 편안하게**, 하지만 유치하지 않게 쓴다.
- 설명은 쉽게 풀되, 학생을 낮춰 보거나 과하게 아기처럼 말하지 않는다.
- 고등학교 2~3학년에게 부적절한 표현, 욕설, 성적/모욕적 표현은 사용하지 않는다.
- 화면당 텍스트 길이는 고정 기준이 없다.
  - 짧아도 괜찮고 길어도 괜찮다.
  - 중요한 것은 길이보다 **가독성, 흐름, 분절, 학습 몰입감**이다.

## UI / UX 원칙

- 전체 인상은 밝고, 차분하고, 둥글고, 학교 수업에 어울리는 톤을 유지한다.
- 모바일과 데스크톱 모두에서 무리 없이 읽히는 반응형 UI를 우선한다.
- 실습 블록은 가능하면 학생의 행동에 대한 시각적 피드백이 분명해야 한다.
- 색만으로 상태를 전달하지 말고, 텍스트/아이콘/레이블도 함께 활용한다.
- 기존 공통 컴포넌트가 있으면 먼저 재사용한다.
- 같은 상호작용이 반복되면 공통 컴포넌트로 분리한다.
- 별도 요청이 없으면 무거운 새 의존성은 추가하지 않는다.

## 구현 원칙

- 먼저 기존 lesson과 공통 컴포넌트를 확인한 뒤 구현한다.
- 반복되는 lesson 패턴은 공통 UI로 분리하고, 단발성 내용은 lesson 내부에서 자연스럽게 구성한다.
- lesson 파일 하나 안에 개념과 실습을 섞어 구성해도 괜찮다.
- 다만 중복 JSX가 많아지면 `src/components/interactive` 또는 적절한 공용 위치로 분리한다.
- 데이터, 예시, 활동 구조는 가능하면 배열/객체로 정리해서 map으로 렌더링한다.
- placeholder, TODO, `준비 중` 같은 문구는 사용자가 명시적으로 원할 때만 남긴다.

## 작업 절차

새 lesson 또는 새 UI 작업을 할 때는 기본적으로 다음 순서를 따른다.

1. 기존 구조와 인접 lesson 파악
2. 필요한 경우 `lessonRegistry` 먼저 수정
3. 해당 module 폴더에 lesson 파일 생성 또는 수정
4. 기존 공통 컴포넌트 재사용
5. 필요하면 새 인터랙션을 공통 컴포넌트로 추출
6. sidebar / navigation / home 흐름이 자연스러운지 점검
7. 결과물이 실제 수업에서 바로 쓸 수 있을 만큼 완성되었는지 확인

## 완료 기준

작업은 아래 기준을 만족하면 완료로 본다.

- 수업 자료가 실제로 사용할 수 있을 만큼 완성되어 있다.
- lesson 내용에 개념과 학생 참여 요소가 모두 들어 있다.
- registry, lesson 파일, navigation 흐름이 서로 어긋나지 않는다.
- 기존 사이트와 톤이 자연스럽게 이어진다.
- 불필요한 placeholder가 없다.

## 검증 원칙

- 이 프로젝트에는 반드시 실행해야 하는 고정 검증 절차가 있는 것은 아니다.
- 별도의 lint/test 규칙이 없더라도, Claude는 스스로 코드 일관성과 구조적 오류 가능성을 줄이는 방향으로 작업한다.
- 시간이 허용되고 코드 변경 폭이 클 때는 `npm run build`를 sanity check로 활용할 수 있다.
- 다만 **검증 명령이 없다는 이유로 결과물 완성을 미루지 않는다.**

## 금지 사항

- 기존 lesson 문장을 그대로 복사하지 않는다.
- 설명만 길고 학생 행동이 없는 lesson을 기본값으로 만들지 않는다.
- 새 lesson을 만들고 registry를 빼먹지 않는다.
- lesson id와 파일 경로 규칙을 깨지 않는다.
- 기존 사이트와 너무 동떨어진 톤이나 UI를 만들지 않는다.
- 학생 대상에게 부적절한 표현을 사용하지 않는다.

## 좋은 결과의 기준

좋은 결과물은 다음과 같다.

- 기존 혜화 데이터랩 흐름에 자연스럽게 이어진다.
- lesson마다 구성이 약간씩 달라도 전체적으로는 개념과 실습이 잘 섞여 있다.
- 실습이 학생에게 실제 행동을 요구한다.
- 답, 피드백, 해설이 lesson 맥락에 맞게 제공된다.
- 학생 주변의 데이터와 상황을 잘 활용한다.
- 다음 lesson 제작에도 재사용할 수 있는 구조와 패턴을 남긴다.
